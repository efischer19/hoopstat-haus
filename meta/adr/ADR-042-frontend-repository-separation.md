## Context

* **Problem:** The Hoopstat Haus monorepo (per ADR-008) currently contains both the Python data backend (apps/, libs/, infrastructure/) and a vanilla HTML/CSS/JS frontend application (frontend-app/). As the project matures, these two concerns have diverged: the backend is a Python-centric data lakehouse with Lambda-based ETL, while the frontend is a static JavaScript application that consumes pre-built JSON and database artifacts via CloudFront. Keeping both in one repository couples their CI/CD cycles, blurs ownership boundaries, and prevents the frontend from benefiting from JavaScript-specific tooling and templates (e.g., `static-js-aws-monorepo-template`).

* **Constraints:**
  - ADR-008 (monorepo structure) was designed for Python applications under `/apps`. The frontend is the only non-Python component and does not use Poetry, Ruff, pytest, or Docker.
  - ADR-035 (CloudFront + OAC) established a single CloudFront distribution that currently serves both Gold data artifacts and frontend app files from the same S3 bucket and `served/` prefix.
  - ADR-038 (Cache Tuning) defined differentiated cache behaviors for data artifacts, frontend assets, and index files — all within one distribution.
  - ADR-040 (Health Dashboard) introduced `health.html` as a pipeline observability tool. This is logically a backend concern (pipeline health monitoring), not a frontend feature.
  - The `deploy-frontend.yml` workflow uses the same OIDC role and S3 bucket as data pipeline deployments.
  - A template repository (`static-js-aws-monorepo-template`) exists for JavaScript-focused static site projects and could serve as the foundation for the new frontend repo.

## Decision

**We will separate the frontend application (`frontend-app/`) into a new, dedicated repository (`hoopstat-haus-frontend`), while retaining the pipeline health dashboard (`health.html`, `health.js`) in this repository as a backend-owned observability component.**

The separation follows these principles:

1. **Clean Ownership Boundary:** This repository becomes a Python-only data backend. The new repository owns all user-facing UI code (data browser, charting, styles).

2. **Health Dashboard Stays:** `health.html` and `scripts/health.js` remain in this repo. The health dashboard visualizes pipeline telemetry generated by the health-aggregator Lambda (also in this repo). It is an operational observability tool, not a consumer-facing feature.

3. **Shared CloudFront Distribution:** The existing CloudFront distribution continues to serve both data artifacts (from this repo's pipeline) and frontend assets (deployed from the new repo). Both repos deploy to the same S3 bucket (`served/` prefix) using separate GitHub Actions workflows. The new repo receives a scoped IAM role or shares the existing OIDC role with path-restricted permissions.

4. **ADR Ownership Transfer:** Frontend-specific ADRs (ADR-019: Vanilla Approach, ADR-036: Chart.js) become the new repository's concern. They are annotated as "Moved" in this repo with a pointer to the new repo. ADR-040 (Health Dashboard) remains here since the health dashboard stays.

5. **Template-Child Relationship:** The new `hoopstat-haus-frontend` repository is created as a template-child of `static-js-aws-monorepo-template`, inheriting its CI/CD patterns, linting configuration, and deployment conventions.

## Considered Options

1. **Separate Repository with Shared CloudFront (Chosen):** Move frontend code to a new repo; both repos deploy to the same S3 bucket and CloudFront distribution.
   * *Pros:* Clean code ownership; single domain/CDN for users; no CORS or cross-origin complexity; frontend gets JS-appropriate tooling; backend repo stays Python-focused; smallest infrastructure change.
   * *Cons:* Two repos deploying to the same S3 prefix requires coordination; shared IAM credentials need careful scoping; cache invalidation must be coordinated.

2. **Separate Repository with Separate CloudFront Distribution:** Each repo gets its own S3 bucket and CloudFront distribution, potentially on different subdomains (e.g., `data.hoopstat.haus` vs `app.hoopstat.haus`).
   * *Pros:* Complete infrastructure isolation; no shared credentials; independent cache invalidation; cleaner blast radius.
   * *Cons:* Requires DNS/subdomain configuration; introduces cross-origin requests from frontend to data API (CORS complexity); users see two different domains; higher infrastructure cost (two CloudFront distributions); more Terraform to manage.

3. **Keep Monorepo, Add JS Tooling:** Keep the frontend in this repo but add JavaScript-specific CI/CD (ESLint, Prettier, etc.) alongside the Python tooling.
   * *Pros:* No repository split; atomic cross-stack changes; single CI/CD pipeline.
   * *Cons:* Polyglot CI/CD adds complexity; frontend changes trigger Python-irrelevant workflows; doesn't align with the template repo strategy; ADR-008 was designed for Python apps.

4. **Git Subtree/Submodule:** Use git subtree or submodule to link the frontend as a nested repository.
   * *Pros:* Preserves git history; can work with both repos independently.
   * *Cons:* Submodules add cognitive overhead and tooling complexity; subtree merges are confusing; neither approach is well-suited for divergent CI/CD pipelines; not compatible with the template-child pattern.

## Consequences

* **Positive:**
  - This repository becomes a focused Python data backend with consistent tooling (Poetry, Ruff, pytest, Docker).
  - The frontend repo can adopt JS-native tooling, testing, and deployment patterns.
  - CI/CD is faster for both repos — frontend changes don't trigger Python linting, and vice versa.
  - The template-child relationship provides a proven foundation for the new repo's CI/CD.
  - The health dashboard remains co-located with the health-aggregator Lambda, maintaining coherent observability ownership.

* **Negative:**
  - Cross-repository coordination is needed for domain/CDN changes and shared infrastructure.
  - The shared IAM OIDC role must be carefully scoped to prevent one repo from affecting the other's S3 objects.
  - ADR history for frontend decisions is split across two repositories.
  - Contributors must clone two repos to work on a feature that spans frontend and backend.

* **Future Implications:**
  - If the shared CloudFront model proves too tightly coupled, migrating to separate distributions (Option 2) is straightforward since the frontend is already in its own repo.
  - The health dashboard could eventually migrate to the frontend repo if the team decides it's more of a UI concern than an ops concern — but the default is to keep it here.
  - New frontend-specific ADRs should be created in the new repository, not in this one.
  - The `static-js-aws-monorepo-template` should be mature enough to support the new repo's needs before the migration begins.
